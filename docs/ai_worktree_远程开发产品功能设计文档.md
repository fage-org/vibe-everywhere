# AI Worktree 远程开发产品功能设计文档

## 1. 文档信息

- 产品名称：vibe-everywhere
- 文档类型：产品功能设计文档（V1）
- 设计目标：面向手机优先、桌面补充的远程 AI 开发工作台
- 参考方向：桌面端交互与信息架构可参考 EnsoAI 的 Git Worktree + AI Agent 工作台思路，但本产品以“远程控制服务器上的 AI worktree”与“移动端可用性”作为核心差异化

---

## 2. 背景与目标

### 2.1 背景

用户在服务器上管理多台主机；每台主机上可能部署多个项目；每个项目当前先只使用一个活跃分支；用户希望通过手机远程查看项目状态、与 AI 对话驱动开发、查看代码文件、查看 Git 修改，并在必要时通过桌面端进行更完整的操作。

### 2.2 产品目标

本产品要解决以下问题：

1. 用户不在电脑前时，仍可通过手机掌握服务器上 AI 开发进度。
2. 用户可以直接在手机上对 AI 下达开发指令，并查看 AI 对代码、终端、Git 的操作结果。
3. 用户可以快速在“服务器 → 主机 → 项目 → 会话 / worktree”之间切换，而不会迷失上下文。
4. 用户可以在手机上完成轻量审查、继续追问、补充需求、查看 diff、批准或终止任务。
5. 桌面端提供更高效率的全量视图和复杂操作，但信息结构与手机端保持一致，降低学习成本。

### 2.3 第一阶段成功标准（个人用户闭环）

第一阶段先面向个人用户，目标不是覆盖团队协作与多角色管理，而是完成一个人从“发起任务”到“审查结果”的完整闭环。

成功标准如下：

- 用户可在手机上完成从连接服务器到发起 ACP 会话的完整闭环。
- 用户可在手机上查看 ACP 会话状态、AI 输出、文件变更、Git diff、终端日([agentclientprotocol.com](https://agentclientprotocol.com/get-started/introduction))- 用户可在桌面端无缝续接同一项目与同一 ACP 会话。
- 用户可在个人环境下安全地让 AI 执行受控开发动作，并对结果进行审查。

## 3. 产品定位

### 3.1 一句话定位

一个以手机优先为核心、基于 ACP 协议连接远程 coding agent 的 AI Worktree 开发工作台，让个人开发者可以随时连接服务器上的项目，与 AI 持续对话并完成代码审查。

### 3.2 核心价值

- 远程：连接服务器统一管理个人开发资源
- 协议化：基于 ACP 与 coding agent 建立标准化会话
- 可追踪：随时掌握 AI 正在做什么、改了什么、卡在哪里
- 可审查：代码、diff、日志、文件改动清晰可见
- 可接力：手机适合发起与审查，桌面适合深度查看与收尾

### 3.3 产品本质

本产品不是单纯 SSH 工具，也不是传统 IDE，而是一个 ACP Client 驱动的远程 AI 开发工作台：

- 以“项目任务视角”而非“命令行视角”组织信息
- 以“ACP 会话 + 项目上下文”作为主入口
- 把文件、Git、终端、AI 对话串成统一任务流

### 3.4 第一阶段边界

第一阶段只解决个人用户场景：

- 一个用户管理自己的服务器、主机和项目
- 一个项目先只对应一个活跃分支
- 暂不考虑多人协作、角色分工、审批流、共享评论等团队能力

## 4. 目标用户与典型场景

### 4.1 目标用户

#### 角色 A：独立开发者

- 管理 1~3 台开发主机
- 经常离开电脑，需要手机远程查看 AI 改代码进度
- 更关注快速发起任务、查看 diff、紧急修正
- 希望在手机和桌面之间无缝接续同一条 AI 会话

> 当前阶段暂不考虑技术负责人、多成员协作、多人共享会话等场景。

### 4.2 典型场景

#### 场景 1：通勤途中发起需求

用户在手机上进入某主机的某项目，通过 ACP 会话对 AI 说“给登录接口加 token 刷新逻辑，并补上测试”，AI 在服务器上执行后，用户可以持续查看会话更新、文件变更、Git diff。

#### 场景 2：中途审查 AI 结果

AI 修改完成后，用户在手机上先看变更文件列表，再点进 diff，重点查看接口、测试、配置文件变更，随后选择“继续优化”或“标记已通过”。

#### 场景 3：任务卡住排障

AI 在执行测试时失败，用户收到通知，进入项目查看日志，发现依赖缺失，于是补充一句“先安装依赖后重试”，继续同一条 ACP 会话。

#### 场景 4：桌面接力收尾

用户在手机上完成发起、跟进与初审，回到桌面端后打开同一项目，恢复同一 ACP 会话，进行更完整的 diff 审查与最终处理。

## 5. 信息架构

V1 建议采用 5 层对象模型：

1. 服务器
2. 主机
3. 项目
4. AI 会话 / 开发任务
5. 代码与变更内容

### 5.1 对象关系

- 一个服务器管理多台主机
- 一台主机包含多个项目
- 一个项目当前默认对应一个活跃分支 / 工作目录
- 一个项目下可以有多条 AI 会话记录
- 每条会话可关联一次或多次开发任务执行

### 5.2 关键实体定义

#### 服务器

表示一组基础设施入口，可包含权限、连接方式、地区等信息。

#### 主机

表示一台可执行 AI 任务的机器，需展示在线状态、资源占用、最近活动。

#### 项目

表示主机上的代码仓库；当前默认一个项目只用一个分支；后续可扩展到多个 worktree / 分支并行。

#### 会话

表示与 AI 的连续对话上下文，应长期保留。

#### 任务执行

表示 AI 对某一轮用户指令的实际操作过程，可能包含：读取文件、编辑文件、运行命令、生成 diff、写入提交说明等。

## 6. 设计原则

### 6.1 手机优先

手机不是桌面缩略版，而是以“决策与审查”为中心：

- 首页先看状态，不先看复杂结构
- 默认展示关键信息摘要，减少横向操作
- 大段代码与大 diff 使用折叠、摘要、按文件查看
- 重要操作要单手可达

### 6.2 任务导向

用户想知道的是：

- AI 正在做什么
- 改了哪些文件
- 是否成功
- 我下一步该做什么

因此首页和项目页都要优先回答这四个问题。

### 6.3 可中断与可恢复

移动场景下，用户频繁中断。因此：

- 退出后再次进入必须快速恢复到上次项目 / 会话
- 长任务要支持后台继续与通知召回
- 日志和对话必须支持断点续看

### 6.4 风险可控

AI 可执行开发动作，但高风险行为要分级控制：

- 只读查看默认开放
- 写文件、运行命令、Git 提交可按策略允许
- 删除文件、强制覆盖、危险命令需二次确认或策略禁用

## 7. 核心功能设计

## 7.1 服务器与主机管理

### 功能目标

让用户先连接到正确的远程环境，并快速识别可用主机。

### 功能点

1. 服务器列表
   - 展示服务器名称、连接状态、主机数量
   - 支持添加 / 编辑 / 删除服务器

2. 主机列表
   - 展示主机名称、在线状态、CPU / 内存占用、当前运行任务数
   - 支持按状态筛选：全部 / 空闲 / 运行中 / 异常

3. 主机详情
   - 显示系统信息、网络状态、磁盘空间、最近项目活动
   - 展示当前正在执行的 AI 任务概览

### 移动端交互

- 首页默认直接显示“最近使用主机 + 运行中项目”
- 主机卡片重点显示：在线状态、项目数、忙闲状态
- 点击主机进入项目列表；长按弹出快捷操作，如“查看终端”“刷新状态”

### 桌面端交互

- 左侧为服务器 / 主机树
- 右侧为主机详情与运行任务面板
- 支持多选、批量刷新、按资源排序

---

## 7.2 项目管理

### 功能目标

帮助用户在多主机、多项目环境中快速定位目标项目。

### 功能点

1. 项目列表
   - 展示项目名、路径、分支名、最后活动时间、当前状态
   - 支持搜索、收藏、最近访问

2. 项目状态摘要
   - 当前分支
   - 未提交文件数
   - AI 当前状态：空闲 / 执行中 / 等待输入 / 失败
   - 最近一次任务结果

3. 项目详情首页
   - 会话入口
   - Git 变更摘要
   - 最近文件变更
   - 终端最近日志

### 移动端交互

- 项目列表项采用卡片式，突出项目名、分支、AI 状态、未提交文件数
- 支持顶部搜索与筛选：运行中、最近活跃、我收藏的
- 项目详情页顶部固定显示状态条：主机 / 项目 / 分支 / AI 状态

### 桌面端交互

- 左栏项目树，中栏 AI 对话，右栏代码 / Git / 日志切换
- 支持多标签打开多个项目

---

## 7.3 ACP 会话与任务面板

### 功能目标

让用户可以像聊天一样发起开发任务，同时以 ACP 的标准会话模型与远程 coding agent 通信，并清楚看到 AI 的实际执行过程。

### 第一阶段的 ACP 核心定位

本产品在第一阶段的核心不是自定义一套 AI 对话协议，而是实现 ACP Client 的关键能力，并把 ACP 会话映射成适合手机和桌面的产品交互。

### ACP 核心能力映射

#### 1. 连接初始化

对应能力：初始化连接、能力协商、版本协商

产品要求：
- 客户端连接远程 agent 时自动完成握手
- 展示 agent 名称、版本、能力、可用会话特性
- 握手失败时给出明确错误与重试入口

#### 2. 鉴权

对应能力：按需认证

产品要求：
- 支持 token / 凭证式认证
- 首次连接时完成认证绑定
- 认证失效时在会话页给出重新认证提示，不让用户陷入空白失败态

#### 3. 会话创建与恢复

对应能力：新建会话、恢复历史会话

产品要求：
- 用户可在项目下新建 ACP 会话
- 用户可恢复最近一次会话，继续同一上下文
- 手机与桌面打开同一项目时，应优先恢复最近活跃会话

#### 4. Prompt 发送

对应能力：发送用户消息 / 任务指令

产品要求：
- 输入框支持自然语言需求输入
- 支持附加当前项目上下文，如当前文件、当前 diff、当前错误日志
- 支持为本轮设置执行约束，如只读、允许改文件、允许运行测试

#### 5. 会话更新流

对应能力：接收 session/update 等增量更新

产品要求：
- 流式显示 AI 进度与中间状态
- 区分文本回复、工具事件、文件改动、命令执行、等待确认等类型
- 移动端默认摘要化显示，避免被大量事件淹没

#### 6. 中断与取消

对应能力：取消当前处理

产品要求：
- 用户可手动暂停或中断当前任务
- 中断后保留上下文，方便继续追问
- 明确告诉用户当前是“已停止”“等待中”还是“失败”

#### 7. 权限请求与用户确认

对应能力：agent 在执行敏感动作前请求许可

产品要求：
- 对写文件、运行命令、Git 操作等给出显式确认模型
- 手机端采用底部弹层或卡片式确认，保证单手可操作
- 支持“本轮允许”与“始终拒绝此类操作”两级策略

### 会话消息类型

- 用户消息
- AI 文本回复
- AI 进度更新
- 文件操作事件
- 命令执行结果
- 权限请求
- 任务完成 / 失败 / 中断状态

### 移动端交互

- 聊天输入框固定底部
- 每轮 AI 响应采用“对话 + 执行事件流”的混合形式
- 长任务默认收起细节，仅展示摘要与进度；点击展开查看详细事件
- 每轮结果下提供快捷按钮：继续优化、解释改动、查看 diff、查看日志、停止任务

### 桌面端交互

- 对话区与事件区可分列展示
- 支持完整查看会话事件流、权限请求历史与执行结果
- 支持从会话快速跳转到对应文件、diff、日志

---

## 7.4 文件浏览与代码查看

### 功能目标

让用户在手机上也能完成轻量代码阅读与定位。

### 功能点

1. 文件树浏览
   - 目录展开 / 收起
   - 搜索文件
   - 最近打开文件

2. 文件查看器
   - 代码高亮
   - 行号显示
   - 跳转到 AI 修改位置
   - 支持只读模式

3. 关注模式
   - 只看本轮改动文件
   - 只看 AI 提及文件
   - 只看失败相关文件

4. 代码解释
   - 选中某段代码，向 AI 提问“这段改了什么 / 为什么这样写”

### 移动端交互

- 默认不展示完整文件树，优先展示“本次变更文件列表”
- 进入文件后支持：字号调节、自动换行、按改动块跳转
- 顶部悬浮小导航显示当前文件名和当前位置
- 代码块支持左右滑动，但尽量提供折行与关键区块聚焦，减少横向滚动痛点

### 桌面端交互

- 文件树常驻左侧
- 支持双栏代码比较与多标签切换

---

## 7.5 Git 变更与审查

### 功能目标

让用户在手机上高效完成 AI 代码审查，这是本产品的关键能力之一。

### 功能点

1. Git 概览
   - 未提交文件数
   - 新增 / 修改 / 删除文件数
   - 最近一次提交信息

2. 变更文件列表
   - 文件名、变更类型、增删行数
   - 支持按文件类型与目录筛选

3. Diff 查看
   - 统一视图优先
   - 可切换为分栏视图（桌面优先）
   - 支持按 chunk 折叠与展开

4. AI 审查摘要
   - 本轮主要改动点
   - 风险提示
   - 测试覆盖情况
   - 可能需要用户确认的决策点

5. 处理动作
   - 让 AI 继续修改某文件
   - 撤销某文件改动
   - 生成 commit message
   - 标记本轮审查通过

### 移动端交互

- 默认先显示“审查摘要”，再进入文件级 diff
- 每个文件顶部展示变更概览与风险标签
- 文件 diff 页面底部固定操作：继续修改 / 撤销 / 解释 / 标记通过

### 桌面端交互

- 左侧文件列表，右侧 diff 详情
- 支持快速键盘导航与多文件并行审查

---

## 7.6 终端与运行日志

### 功能目标

让用户在不直接操作复杂终端的前提下，仍能看到 AI 执行命令和测试结果。

### 功能点

1. 终端日志流
   - 展示 AI 执行命令
   - 展示标准输出与错误输出
   - 高亮错误、警告、测试摘要

2. 预设命令
   - 安装依赖
   - 运行测试
   - 构建项目
   - 查看 git status

3. 日志过滤
   - 全部 / 命令 / 错误 / 测试结果

4. 异常处理
   - 一键复制错误
   - 让 AI 根据错误继续修复

### 移动端交互

- 日志默认按时间流展示
- 错误块自动置顶摘要显示
- 提供“基于此错误继续修复”快捷入口

### 桌面端交互

- 支持完整终端视图与命令重跑
- 后续版本可开放半交互式终端

---

## 7.7 通知与异步召回

### 功能目标

移动端必须让用户在离开页面后仍能被关键结果召回。

### 功能点

1. 通知类型
   - 任务完成
   - 任务失败
   - 等待用户确认
   - 测试通过 / 测试失败

2. 通知层级
   - 系统推送
   - 应用内消息
   - 项目内红点

3. 召回动作
   - 点通知直达对应项目与会话
   - 自动定位到最新结果

### 移动端交互

- 支持按项目订阅通知
- 支持“仅通知失败和待确认”模式，避免被频繁打扰

---

## 7.8 权限与安全控制

### 功能目标

保证远程 AI 执行可控，不让手机端的便捷性变成风险来源。

### 功能点

1. 权限分级
   - 只读模式
   - 可编辑文件
   - 可运行安全命令
   - 可执行 Git 提交
   - 高危操作禁用

2. 敏感操作确认
   - 删除文件
   - 覆盖大量文件
   - 执行高风险 shell 命令

3. 审计记录
   - 用户下达了什么命令
   - AI 执行了什么操作
   - 哪些文件被改动

4. 登录与设备安全
   - 生物识别解锁
   - 会话超时
   - 敏感页面二次验证

---

## 8. 手机端重点交互设计

## 8.1 全局导航

建议采用底部 4 Tab：

1. 首页
2. 项目
3. 通知
4. 我的

### 首页

聚焦“现在最值得你关注的东西”：

- 运行中的 AI 任务
- 最近访问项目
- 待你确认的变更
- 失败任务告警

### 项目

按“主机 > 项目”浏览，也支持直接搜索项目。

### 通知

集中查看任务结果与待确认事项。

### 我的

管理服务器连接、偏好设置、安全策略。

## 8.2 首页结构建议

从上到下：

1. 顶部状态区
   - 当前在线主机数
   - 运行中任务数
   - 待处理告警数

2. 继续工作卡
   - 最近一个项目
   - 当前会话摘要
   - 一键继续对话

3. 运行中任务列表
   - 项目名
   - 当前步骤
   - 已运行时长
   - 查看详情

4. 待审查列表
   - 项目名
   - 变更文件数
   - 最近测试结果

## 8.3 项目详情结构建议

顶部：
- 项目名
- 主机名
- 分支
- AI 状态

中部采用二级导航：
- 会话
- 变更
- 文件
- 日志

默认落在“会话”页，因为用户核心诉求通常是继续任务。

## 8.4 会话页交互建议

每轮 AI 返回拆成 3 层：

1. 结果摘要
2. 操作事件流
3. 可展开的详细输出

这样手机上不会被大段日志淹没。

## 8.5 Diff 页交互建议

- 先按文件展示
- 点击进入文件级 diff
- 大文件默认只显示变更块
- 提供“AI 总结这个文件改动”的按钮

## 8.6 输入框设计

输入框需要支持：

- 文本输入
- 快捷指令模板
- 追加上下文（如当前错误日志、当前文件、当前 diff）

推荐预置快捷操作：

- 修复当前错误
- 继续完成未完成部分
- 解释这次改动
- 为这轮修改补测试
- 只修改这个文件

## 9. 桌面端设计建议（参考 EnsoAI，但做差异化）

EnsoAI 的价值在于 Git Worktree 管理、多 Agent 切换、Git 面板、代码编辑器和轻量 IDE 桥接。其 README 将产品定位为“Git Worktree + AI Agent 协作工作台”，并强调每个 worktree 拥有独立 AI 会话、可视化 Git 面板、轻量编辑器和 Open In IDE 能力。([github.com](https://github.com/J3n5en/EnsoAI/blob/main/README.zh.md))

本产品桌面端可参考其“工作台”思路，但要强化远程场景：

### 建议桌面布局

- 左侧：服务器 / 主机 / 项目树
- 中间：AI 会话与任务流
- 右侧：代码 / Git / 日志面板

### 建议差异化点

1. 更强的远程多主机管理
   - EnsoAI 更偏本地 worktree 工作台；本产品应突出服务器、主机、项目三级管理。([github.com](https://github.com/J3n5en/EnsoAI/blob/main/README.zh.md))

2. 更强的移动协同一致性
   - 同一会话在手机与桌面无缝续接

3. 更强的“任务回放”能力
   - 可以回看 AI 每一步读了什么、改了什么、跑了什么

4. 更明确的安全策略中心
   - 管理哪些主机、哪些项目允许 AI 执行到什么权限级别

## 11. 关键用户流程

## 11.1 流程 A：手机发起 AI 开发任务

1. 打开 App 进入首页
2. 点击最近项目或搜索项目
3. 进入项目详情页
4. 在会话页输入需求
5. 选择执行权限范围
6. 发送给 AI
7. 查看执行状态与中间结果
8. 收到完成通知
9. 查看 diff 并决定继续修改或通过

## 11.2 流程 B：手机审查 AI 结果

1. 收到任务完成通知
2. 进入对应项目
3. 查看 AI 生成的摘要
4. 查看变更文件列表
5. 打开关键文件 diff
6. 对某个文件追加“继续优化”要求
7. AI 二次修改
8. 最终标记审查通过

## 11.3 流程 C：桌面接力收尾

1. 在桌面端打开同一项目
2. 自动恢复最近会话与变更上下文
3. 全量查看 diff 与日志
4. 必要时进一步编辑
5. 执行提交或推送

## 12. 第一阶段范围与非目标

## 12.1 第一阶段必做（个人用户闭环）

- 多服务器 / 多主机 / 多项目管理
- 基于 ACP 的连接、初始化、认证、会话创建、会话恢复
- 手机端 AI 对话发起与跟进
- ACP 流式状态展示与任务中断
- 文件查看
- Git 变更查看
- 终端日志查看
- 基础权限控制与敏感动作确认
- 手机通知召回与跨端续接

## 12.2 第一阶段可选增强

- AI 自动总结本轮改动
- 生成 commit message
- 最近错误一键续修
- 项目收藏与最近访问

## 12.3 第一阶段暂不做

- 多人协作与多角色权限体系
- 共享会话、团队审查流、审批流
- 一个项目内多分支 / 多 worktree 并行完整管理
- 手机端复杂代码编辑
- 三方协作评论系统
- 完整 IDE 级别的调试体验
- 高复杂度 merge conflict 三栏编辑

## 15. 风险与设计注意事项

1. 手机端阅读代码天然吃力
   - 解决：以摘要、文件级导航、改动块聚焦为主，不追求全 IDE 体验

2. 长日志和长 diff 信息过载
   - 解决：默认摘要化，按需展开

3. 多主机多项目容易迷失上下文
   - 解决：全局统一显示当前主机 / 项目 / 分支 / 会话状态

4. AI 执行动作存在风险
   - 解决：分权限模型 + 高风险确认 + 审计记录

5. 手机和桌面体验可能割裂
   - 解决：统一对象模型与导航语义，保证跨端无缝续接

## 16. 产品总结

本产品的本质，不是把桌面 IDE 缩到手机里，而是把“远程 AI 开发协作”重构成一个适合移动场景的任务工作台。

手机端负责：
- 发起任务
- 跟进进度
- 查看变更
- 审查结果
- 继续对话

桌面端负责：
- 多项目并行管理
- 更高效的代码与 diff 审查
- 深度开发与收尾操作

这样既继承了 EnsoAI 这类产品在 AI worktree 工作流上的优势，也建立了“远程多主机 + 手机优先”的明确差异化。
